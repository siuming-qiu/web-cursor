# Roadmap —— 从可靠 Agent 到大型项目自治

> Status: active design roadmap，除“当前基线”中明确标记完成的能力外，均非当前实现。
> Revised: 2026-07-15。
> 产品目标:把 Web Cursor 从“会生成并运行 React 项目”升级为“能够可靠完成多文件编码任务的 Agent”。
> 学习目标:手写理解 run lifecycle / eval / memory / retrieval / observability / multi-agent / MCP，不引入 LangChain 系框架。

## 1. 定位与北极星

本路线把产品目标从“小型 React 生成沙箱”升级为“能在大型 React 项目上工作的编码 Agent”。它推翻了 `REQUIREMENTS.md` v0.2 中“固定依赖、非完整工程”的部分非目标；P-1～P5 可先解决当前产品的 Agent 可靠性，正式进入 P6 的大型项目与后台执行架构前，必须先在下一版需求文档中确认这次产品转向。

### 1.1 最终验收靶子

Agent 能在一个 100+ 文件的真实模板项目（例如 admin dashboard）中：

1. 在不知道目标文件路径时定位相关实现。
2. 完成一个跨 3～5 个文件的功能修改。
3. 不覆盖用户并发修改，不把用户写错的代码归因给 Agent。
4. 通过项目契约明确声明的 typecheck、build 和浏览器预览验证。
5. 发生 Agent 自己引入的错误时自动修复；达到上限或遇到冲突时明确停止并报告。
6. 支持真正停止、过程回放和撤销本次 Agent 修改。

### 1.2 核心指标

| 指标 | 说明 |
|---|---|
| 任务成功率 | 固定任务集最终通过全部验收的比例，作为北极星 |
| 错误归因准确率 | Agent 错误会进入自修；用户错误绝不自动触发 Agent |
| 非预期写入率 | 停止后写入、冲突覆盖、失败 mutation 触发刷新都必须为 0 |
| 首次成功耗时 | 从用户发送需求到第一次完整验证通过 |
| 修复轮次 | 首次验证失败后，到最终通过所需的 Agent 轮次 |
| 工具与上下文成本 | tool 调用数、输入/输出 token、读写文件字符数 |

## 2. 当前基线（2026-07-15，`ff4613e`）

### 2.1 已完成

- Postgres 持久化项目、项目文件、会话和 transcript；项目文件仍是代码权威来源。
- 手写服务端 Agent loop，并支持 `list_files`、`search_text`、`read_file`、`write_file`、删除、重命名和 `run_preview`。
- `b0b17c6` 已接通首屏浏览器运行错误反馈：Agent 修改后的预览错误可以回填并继续修复。
- `ff4613e` 已实现后端 `search_text` v1：在已保存项目文件中做有界、大小写敏感、单行字面量搜索。
- WebContainer 能安装依赖、启动 Rsbuild dev server，并在 iframe 中返回首屏 `RENDER_OK` / `RUNTIME_ERROR`。

### 2.2 已核实的断裂点

| # | 断裂点 | 当前后果 |
|---|---|---|
| 1 | 失败的文件 mutation 仍可能发送通用 `files_changed` | 前端会刷新甚至预览一次实际未发生的修改 |
| 2 | “停止”只停止前端消费，请求和服务端 loop 没有 run 级取消 | 停止后旧 Agent 仍可能执行工具；新旧 run 可能交叉写入 |
| 3 | 用户保存、Agent 搜索和 Agent 写入之间没有同步屏障与 revision 条件 | Agent 可能读取旧 DB 内容；用户与 Agent 之间是 last-write-wins |
| 4 | Preview 没有携带修改来源和项目 revision | 无法严格证明错误由当前 Agent 修改导致 |
| 5 | transcript 全量回放，旧 `read_file` / `write_file` 全文长期留在上下文 | 长会话成本持续增长，旧源码还可能被误当成当前事实 |
| 6 | `write_file` 只能整文件覆盖，`list_files` 无界且修改流程强制先调用 | 小改动浪费 token；大项目文件列表本身会污染上下文 |
| 7 | WebContainer 每次预览全量 mount、执行 `npm install` 并重启单例 dev server | 多轮修复耗时高，用户预览和 Agent 预览可能互相取消 |
| 8 | 运行桥只观察首屏 1 秒，缺少受控 console、延迟错误和分层验证 | 异步错误可能漏报；只有完整预览，没有更快的 typecheck/build 层 |
| 9 | 没有一等 AgentRun、trace、changeset 和 checkpoint | 无法可靠取消、恢复、评估和撤销一次完整任务 |

这些问题的优先级高于继续扩展 LSP、语义检索或多 Agent。

## 3. 目标架构原则

### 3.1 当前阶段保持单一代码真相源

```text
Postgres project_files      代码权威来源
        │
        ├── 服务端文件工具  list / search / read / edit / write
        │
        └── 浏览器同步      Editor + WebContainer preview
```

在性能和能力数据证明必要之前，不为每个项目额外维护一个持久后端文件系统，也不把 Git 提前升级为代码权威来源。否则会同时出现 DB 与 workspace 两套真相、双向同步、生命周期、成本和安全问题。

### 3.2 AgentRun 先于持久 workspace

目标关系是：

```text
Project ── ProjectFile（带版本）
   │
Conversation ── AgentRun ── ToolCall / ToolResult
                    ├── ChangeSet
                    └── PreviewRun（绑定来源与项目版本）
```

这是概念关系，不是数据库字段契约。具体字段、状态枚举和迁移必须在对应变更设计中明确，禁止实现时猜 schema 或用默认值吞掉未知状态。

### 3.3 执行域逐步演进

```text
A. LLM 代理域        Next.js 服务端；持 key、推理、工具编排
B. 浏览器编排域       编辑器、用户动作、Agent 过程显示、客户端工具握手
C. WebContainer      当前即时预览与浏览器运行反馈环境
D. 后端执行沙箱       条件性能力；用于 headless typecheck/build/preview
```

- P0～P5 继续用 A/B/C 完成可靠 Agent 闭环。
- D 域不是默认前提；只有 P6 的大型项目基准证明需要时才引入。
- 即使引入 D 域，不可信项目代码也绝不能运行在 Next.js 应用进程中。

## 4. 路线纪律

1. **因果关系优先**：只有绑定当前 AgentRun 和当前项目版本的错误，才能触发自动修复。
2. **取消必须真实**：停止后的 run 不得再开始新工具或产生文件写入；不能只隐藏 UI。
3. **冲突必须暴露**：版本不一致、替换目标为 0 个或多个时返回明确错误，不做模糊修复或整文件覆盖兜底。
4. **先 baseline，再优化**：每项上下文、检索、工具或执行环境升级都要能在同一任务集上比较。
5. **先简单工具，再高级索引**：字面搜索能解决的问题，不引入 LSP、向量库或知识图谱。
6. **单一真相源**：任何阶段都必须明确当前代码权威来源；不长期维护隐式双写。
7. **向后兼容**：现有项目、会话、手动编辑和 Agent 自修闭环不能因升级失效。

## 5. 迭代路线

### P-1 —— 当前版本 Baseline Snapshot

**目标**：在改变 Agent 行为前，为 `ff4613e` 留下一份可复查的起点；允许当前版本在安全性用例中出现预期失败。

工作项：

- 使用隔离 fixture 运行最小只读检索任务，并人工记录当前多文件修改、自修、用户错误、停止重发和保存后搜索行为。
- 保存代码版本、模型、prompt/fixture 版本、tool 顺序、结果和已知失败；不把真实用户项目作为固定测试数据。
- 安全性用例只在可丢弃项目中执行，避免为了测量已知缺陷而破坏用户数据。

这一步是快照，不要求先搭完整 eval 平台；P0 完成后的 P1 才建立正式、可重复的回归基线。

### P0 —— Agent Run 一致性与来源隔离

**目标**：先确保 Agent 不乱写、不串线、不会修复用户自己造成的错误。

工作项：

- 修复失败 mutation 仍发送 `files_changed` 的确定性 bug；只有成功修改才发送文件变化事件。
- 定义 AgentRun 生命周期和唯一标识，让消息、工具调用、文件变化和 Preview 都能归属到同一次 run。
- 持久化最小、权威的 run 协调状态和项目写入 lease，至少能跨请求和多实例判断当前写入 run 与取消状态；P5 再扩展完整 trace、恢复和历史字段。
- 实现真正取消：浏览器中止请求，服务端在开始下一工具前检查取消状态，取消后拒绝新的 mutation。
- 同一项目同一时间只允许一个写入型 AgentRun；重发、恢复和多会话不能交叉写。
- 发送聊天前处理未保存草稿并等待保存落库；同步期间不能让 Agent 基于旧 DB 开始任务。
- 第一版在 Agent 写文件期间锁定用户文件 mutation；revision/CAS 完成后再考虑开放并发编辑。
- 为修改、删除和重命名建立版本条件；不匹配时显式返回冲突，要求重新读取。
- 消除 `write_file` 当前“存在则覆盖、不存在则创建”的隐式 upsert 语义：新建必须确认目标不存在，完整覆盖已有文件必须携带已读取版本；冲突不得退化为无条件覆盖。具体参数由变更契约定义。
- Preview 绑定“用户或 Agent 来源、所属 run、项目版本”；只有当前 Agent 来源且版本一致的错误进入自修。

验收：

- 文件工具失败时，前端不刷新文件、不触发预览。
- 连续“发送 → 停止 → 再发送”不会出现旧 run 的晚到写入或 tool result。
- 用户保存唯一 marker 后立即让 Agent 搜索，必须稳定命中最新内容。
- 用户与 Agent 试图修改同一文件时，不出现静默覆盖。
- Agent 写错代码会自动修；用户手动写错只展示错误，不自动唤醒 Agent。

### P1 —— Baseline Eval v0 与最小 Trace

**目标**：让后续每次 Agent 迭代都有可比较的数字，而不是凭体验判断“变聪明了”。

工作项：

- 建立 8～10 个版本化固定任务，fixture 与真实用户项目隔离。
- 覆盖未知路径定位、跨文件修改、多匹配、无匹配、特殊字面量、首屏错误自修、用户错误不自修、停止重发、保存后立即搜索。
- 从 transcript 提取最小 trace：任务、模型、代码版本、fixture 版本、tool 顺序、轮次、耗时和断言结果。
- 报告不保存完整源码、敏感附件或密钥；只保留评分所需摘要。
- 先采用浏览器驱动的人工/半自动完整任务；后端沙箱完成后再升级为 headless eval。

硬性断言：

- 只读任务不得产生 mutation。
- 每个 tool call 必须恰好有一个闭合结果。
- 未知路径任务必须先定位再读取，修改前必须读取当前文件。
- 用户手改错误的自动修复次数必须为 0。
- 停止确认后的文件写入次数必须为 0。

首次运行只记录成功率、耗时、轮次和 token，不凭空制定性能门槛；有 baseline 后再冻结回归阈值。

### P2 —— 上下文预算与工具选择效率

**目标**：长对话和大项目下，Agent 仍能继续工作，并始终以当前代码为事实来源。

工作项：

- 定义明确输入预算，按完整 user turn 裁剪；assistant tool call 与 tool result 永远成对保留。
- “完整 turn”必须依据 P0 的真实人类请求与 AgentRun 边界识别，不能只看 message role；内部 `preview_feedback` 即使存成 user role，也不代表新的用户请求。
- 保留当前请求、未闭合调用和最近有效约束；旧文件全文从上下文中淘汰，需要时重新 `search_text + read_file`。
- 对 npm/build 日志做有界、确定性的错误摘要，保留原始诊断引用但不把无限日志塞给模型。
- 取消“修改项目必须先无界 `list_files`”的提示词要求：已知标识符时允许直接搜索。
- 为文件列表设计明确的过滤与分页契约；未知参数直接报错，不静默补默认业务语义。
- 记录每轮上下文组成和 token 使用，供 P1 eval 比较。

验收：

- 30～50 轮会话仍能合法重建 tool pairing，不因旧源码全文持续增长而失败。
- 1000 文件项目定位唯一文本时，不需要先把全部路径放进上下文。
- 历史 `read_file` 结果不会覆盖 DB 中更新后的当前内容。

### P3 —— 安全的精确编辑与 ChangeSet

**目标**：小范围修改不再整文件重写，并通过版本条件防止覆盖用户新改动。

工作项：

- 增加精确 `edit_file`：携带目标文件版本、旧文本和新文本。
- 旧文本必须恰好匹配一次；0 次、多次、版本变化都返回明确、可诊断的冲突。
- 禁止模糊匹配、行号猜测和失败后自动退化为整文件覆盖。
- `write_file` 继续负责新建文件和明确的完整重写，不强行让所有场景使用 patch；新建必须验证路径不存在，覆盖必须满足版本条件。
- 一次 AgentRun 的成功文件变化形成 ChangeSet，为 trace、预览归因和撤销提供基础。

验收：

- 单行修改的工具参数不再包含整份文件正文。
- 文件被用户修改后，Agent 的旧版本 edit 必须失败并重新读取。
- 替换文本重复出现时，不允许随便选择第一个位置。
- 创建新文件与大规模重写仍能通过 `write_file` 正常完成。

### P4 —— 分层验证与 Preview v2

**目标**：Agent 用最快的可靠反馈修代码，而不是每次都重新安装并等待完整页面。

验证顺序：

```text
静态契约检查 → typecheck（有权威脚本时）→ build → 浏览器预览
```

工作项：

- `build` 只执行项目契约明确声明的构建脚本。
- `typecheck` 只有在模板契约要求或项目显式声明脚本后才能执行，禁止猜测命令和依赖。
- 为编译、构建、运行时错误定义严格且有界的 ToolResult；未知 provider 结果暴露为可诊断错误。
- 捕获受控 console 和延迟运行错误，限制条数、单条长度和总大小。
- 延迟错误必须绑定原 AgentRun、PreviewRun 和项目版本；不属于当前 Agent 修改时只展示。
- Preview 契约必须定义有界观察期和明确终止点；观察期结束后的晚到错误只能展示，不能重新唤醒已经终止的 AgentRun。
- 记录 mount、install、build、server-ready、runtime 各阶段耗时，再决定依赖缓存和增量 mount。
- 依赖声明未变化时复用安装结果；增量同步必须正确处理删除和重命名，不能残留脏文件。

验收：

- 类型或构建错误无需进入浏览器即可返回 Agent 修复。
- 1 秒以后出现的 Agent 异步错误可以进入一次受控修复。
- 相同的用户手改错误只展示，不触发 Agent。
- 依赖未变化时，第二次纯源码修改的验证耗时明显低于首次运行，并由 eval 记录。

### P5 —— 持久 AgentRun、Observability 与回滚

**目标**：一次 Agent 任务可以被可靠观察、显式恢复和撤销，run 状态不再等同于一条 SSE 连接。

工作项：

- 在 P0 最小协调状态之上，扩展并持久化完整 run 生命周期、轮次、工具摘要、错误原因、耗时和 token 使用。
- SSE 只负责传输过程；断线重连可以回放已持久化事件，并从最后一个已闭合工具边界显式恢复，不能重复执行 mutation。
- 客户端工具调用在等待浏览器执行时进入明确等待态；没有执行环境时不得伪装成功。
- 提供折叠式 trace：搜索、读取、编辑、验证和修复过程可回放，但默认不展示完整源码参数。
- 基于 P3 ChangeSet 实现“一键撤销上一次 Agent 修改”；只撤销 Agent 变化，不覆盖之后的用户修改。
- 工具调用执行与结果写入分别具备幂等边界：同一 tool call 被重放时不能再次执行 mutation，晚到或重复 result 必须明确拒绝或返回已有闭合结果。
- 在引入持久任务执行器以前，不承诺浏览器断线后 Agent 仍在后台无限继续；无人值守运行属于 P6 的独立决策。

验收：

- 刷新页面后能恢复同一次 run 的已完成过程和当前状态。
- 同一个 mutation tool call 被重放时，不会产生第二次文件修改。
- 同一个 tool result 重复或晚到提交时，不会追加第二个闭合结果，也不会再次恢复 Agent loop。
- 用户在 Agent 完成后又修改文件时，撤销不得覆盖用户的新内容。
- trace 能回答“Agent 为什么改这个文件、在哪一步失败、修了几轮”。

### P6 —— 大型项目与后台执行环境决策

**目标**：用真实基准选择执行环境，而不是预设“持久 workspace 一定更快”。

使用 P1～P5 的 eval 在 100+ 文件模板上测量：

- 文件同步体积与耗时。
- WebContainer mount/install/build/preview 各阶段耗时和失败率。
- 浏览器断线、休眠和资源限制对任务成功率的影响。
- 是否存在必须无人值守执行的 headless eval 或后台 AgentRun。

持久 Agent 调度器与项目代码 workspace 是两项独立决策：后台 LLM/tool loop 不要求持久项目文件系统；反过来，持久 workspace 也不会自动解决任务调度、重试和租约。

编排层先按证据选择：

| 方案 | 适用条件 |
|---|---|
| 请求内执行 + 显式恢复 | 用户保持页面在线已满足成功率；维持最简单运行模型 |
| 持久任务执行器 | 无人值守任务或断线后继续执行已成为真实需求；必须同时设计租约、重试和幂等 |

代码执行层再独立选择：

| 方案 | 适用条件 |
|---|---|
| DB + WebContainer | 浏览器执行已满足任务成功率和性能目标；保持最简单架构 |
| DB + 一次性后端沙箱 | 需要 headless typecheck/build/preview，但不需要长期保存运行环境 |
| DB + 可回收缓存 workspace | 重复物化和依赖安装已成为主要瓶颈，需要受控复用 |
| 持久 Git workspace | 只有长期后台任务、分支隔离或多 Agent 已成为真实需求时考虑 |

引入任何后端执行环境时，配额、超时、CPU/内存/进程限制、网络 allowlist、无生产密钥和独立产物 origin 必须同批上线。Provider 价格与免费额度必须在实施当天重新核实。

### P7 —— 条件性高级检索与长期记忆

**目标**：只补 baseline 证明字面搜索解决不了的检索失败。

升级顺序：

1. 改善 `search_text` 工具形状：路径范围、大小写选项、结果文件分布等，均由失败样本驱动。
2. import 依赖图与 TypeScript 符号索引，用于精确“定义/引用/影响范围”问题。
3. 只有符号检索仍不能解决自然语言概念定位时，才考虑 Postgres 向量检索。
4. 只有跨会话重复错误达到可测规模时，才建立“错误签名 → 已验证修法”的经验库。

不得为了学习主题预设必须使用 LSP、tree-sitter、ts-morph、pgvector 或知识图谱；每一层都必须在同一 eval 上证明增量收益。P7 可以在 P1/P2 出现检索失败证据后启动；如果具体实现依赖后端 workspace，则仍须先通过 P6 的执行环境决策。

### P8 —— 条件性多 Agent 与协议化

**目标**：单 Agent 在大型任务上出现可量化瓶颈后，再引入并行协作。

完整目标架构、上下文共享、Agent 通信、可观察 UI 与写入隔离方案见 [`multi-agent-architecture.md`](./multi-agent-architecture.md)。

前置条件：

- 单 Agent 的 run、revision、ChangeSet、trace 和 eval 已稳定。
- 任务确实能按模块拆分，并且并行收益高于上下文与合并成本。
- 执行环境能提供隔离分支或等价的变更隔离。

候选能力：

- planner 拆解任务，worker 在隔离变更集上执行，reviewer 检查 diff 与验证结果。
- 汇聚阶段使用明确冲突协议，不靠最后写入获胜。
- 内部工具只有在需要被外部 Agent 或第三方客户端复用时才 MCP 化。
- SSE 事件协议只有在真实互操作需求出现时再对照 AG-UI 重审。

## 8. 测试与 Eval 原则

- 确定性测试只保护真实不变量：schema 严格性、revision 冲突、tool pairing、取消后不写入、来源归因和幂等性。
- LLM eval 有随机性和调用成本，适合手动触发或定时执行，不作为每次本地修改的强制测试。
- Preview、停止重发、用户与 Agent 并发修改必须有集成或浏览器级验证，纯单元测试不足以证明闭环。
- UI 文案、简单展示或机械类型同步不新增仪式性测试。
- 每次改变 system prompt、工具描述、上下文裁剪或检索策略，都要保留变更前后同一 fixture 的报告。

## 9. 明确暂不纳入

- 语音 ASR/TTS、模型训练与推理原理：与本产品 Agent 工程主线解耦。
- Nest：本项目继续使用 Next 全栈，不引入第二后端框架。
- Milvus、ElasticSearch、Neo4j：当前数据规模不需要额外基础设施。
- Redis 短期记忆：短期会话事实继续由 Postgres transcript 管理。
- 默认持久 workspace：P6 以前不建设。
- 为了“显得像 Agent”而增加多 Agent：P8 前不建设。
